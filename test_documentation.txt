3) Test Settings
Django tests:
- pytest uses: `orchestrator_backend.settings_test`
- Dedicated test settings force SQLite so tests never connect to the real Supabase/Postgres database.

Services tests:
- pytest uses: `services/pytest.ini`
- This isolates service tests from Django pytest settings defined at the project root.
- External APIs are mocked inside the tests.

4) RabbitMQ Upload Flow Impact
- `documents_upload` is now asynchronous.
- Expected upload response: HTTP 202 with `status=Processing`.
- Django tests validate queue publishing by mocking `_enqueue`.
- Django upload tests no longer expect synchronous `/ingest` and `/index/upsert` calls.

Files updated for this flow:
- orchestrator_backend/gateway/tests/test_documents_api.py
- orchestrator_backend/tests/integration/test_documents_endpoints.py

5) Local Installation
From project root (`microservices`):
- `pip install -r orchestrator_backend/requirements.txt`
- `pip install -r orchestrator_backend/requirements-dev.txt`
- `pip install -r services/requirements.txt`
- `pip install pytest pytest-cov`

6) Local Commands
Run all Django tests:
- `python -m pytest orchestrator_backend/gateway/tests orchestrator_backend/tests -v`

Run Django tests with coverage:
- `python -m pytest orchestrator_backend/gateway/tests orchestrator_backend/tests --cov=orchestrator_backend/gateway --cov-report=term-missing --cov-report=html --cov-report=xml -v`

Run only the upload-related Django tests:
- `python -m pytest orchestrator_backend/gateway/tests/test_documents_api.py orchestrator_backend/tests/integration/test_documents_endpoints.py -v`

Run all services tests:
- `cd services`
- `python -m pytest tests -c pytest.ini -v`

Run services tests with coverage:
- `cd services`
- `python -m pytest tests -c pytest.ini --cov=. --cov-report=term-missing --cov-report=html --cov-report=xml -v`

7) GitHub Actions CI
Workflow file:
- `.github/workflows/ci.yml`

Pipeline behavior:
- Triggered on every push
- Triggered on every pull request
- Can also be launched manually from GitHub with `workflow_dispatch`

Job 1: `django-tests`
- Installs Python 3.12
- Installs Django dependencies and test dependencies
- Runs Django unit and integration tests with pytest
- Generates coverage in terminal output, `coverage.xml`, and `htmlcov/`
- Uploads the coverage files as GitHub Actions artifacts

Job 2: `services-tests`
- Installs Python 3.12
- Installs FastAPI services dependencies
- Runs `services/tests`
- Uses `services/pytest.ini` to avoid interference from Django pytest settings
- Generates services coverage in terminal output, `services/coverage.xml`, and `services/htmlcov/`
- Uploads the coverage files as GitHub Actions artifacts

8) Environment Variables in CI
For this pipeline, no real production secrets are required.
The workflow injects safe dummy values for variables such as:
- `SUPABASE_URL`
- `SUPABASE_KEY`
- `SUPABASE_SERVICE_KEY`
- `PINECONE_API_KEY`
- `HF_TOKEN`
- `RABBITMQ_URL`

Why this is safe:
- External systems are mocked by the test suite.
- Django test settings use SQLite, not the production database.
- Services tests patch network-facing dependencies before importing the apps.

9) What You Must Do in GitHub
1. Push your code to GitHub.
2. Open the repository on GitHub.
3. Go to the `Actions` tab.
4. On the next push or pull request, the workflow will run automatically.
5. Open the workflow run to see:
   - install logs
   - pytest results for both jobs
   - pass/fail status
   - uploaded coverage artifacts

10) What You Do Not Need Yet
- No GitHub Secrets are required for this first CI pipeline.
- No live database service is required.
- No live RabbitMQ service is required.
- No live Supabase, Pinecone, or Hugging Face credentials are required.
